2The one habit

Describe what you want in plain language. The matching skill triggers on its own, and when it finishes it offers the next step. Say yes, and the chain flows:

You stay in control — it suggests the next step, you decide. Two reflexes are always on: /review asks a safety-gate question ("does this touch auth or data?") and runs /security-review when yes; and /diagnose takes over the moment something breaks. Any point is a valid start — a bug report jumps straight to debug, "review my branch" straight to review.

3Your first change, start to finish

A real example. The italic line is what you type; below it is what happens.

You type“Help me add an endpoint that returns a customer's orders.”
/grill-with-docs fires. It asks the sharp questions — “Only active orders, or archived too? Paginated? What does an empty result return?” — and records agreed terms in CONTEXT.md. Then: “Clear enough — want me to build it with /tdd?”
You type“Yes, build it.”
/tdd writes one failing test (customer sees their orders newest-first), makes it pass, then the next — one behaviour at a time. When green: “It's a customer-facing endpoint — prove it end-to-end with /functional-test?”
You type“Yes.”
/functional-test writes an API test that hits the real endpoint: an authenticated request returns that customer's active orders, newest-first, and a request with no token is rejected. Green. “Want me to /review before merge?”
You type“Review it.”
/review checks the diff against what you agreed, then asks the safety gate: “This endpoint reads member data — I'll run /security-review before merge.”
You type“Go ahead.”
/security-review finds the real problem: the endpoint never checks that the caller owns the customer — any signed-in user could read anyone's orders (an IDOR). It recommends an ownership check and a failing test for the unauthorized case. You fix it with /tdd; green → merge.

Five plain sentences, and the agent walked the whole loop — proving the feature end-to-end and catching a data-exposure bug before it shipped. You never looked up a command.

5Good to know

Do I have to run setup or create docs?

No. Nothing is required. CONTEXT.md and ADRs are created only if they'd help, and never for trivial changes.

What if I don't know which command to use?

Don't — just describe the task. If nothing fires, type the closest phrase (e.g. "review this") and it will.

Touching auth or data?

/review asks a safety-gate question before merge and runs /security-review when the answer is yes. In data-sensitive work (e.g. anything with PHI/PII) that pass is expected, not optional.

Will it change my code without asking?

It proposes and offers next steps; you approve each hop. The chain guides, it doesn't run away.

Can I make a skill fit how my team works?

Yes — each skill is a single SKILL.md in skills/<name>/. Open it, edit the wording, save. To remove one, delete its folder and its line in plugin.json.

All skills all 14 · one plugin

The complete toolkit in the single engineering-skills plugin, grouped by phase — planning, triage, architecture, and handoff, plus the everyday skills above. They all install together; you only invoke what a task needs.

Setup · once per repo
/setup-skills
writes: CLAUDE.md, docs/agents/*.md
Records this repo's issue tracker (GitHub, Jira, or local markdown), where PRDs/ADRs live (in-repo or a central docs repo), the triage label vocabulary, and the CONTEXT.md layout.
Use it once per repo before anything else — every other skill reads what it writes. Re-run only to switch trackers.
Align
/grill-me
Interviews you relentlessly about a plan or design, one question at a time, resolving every branch of the decision tree.
Use it to stress-test any non-code plan before committing to it.
/grill-with-docs
updates: CONTEXT.md, docs/adr/
The same grilling, for code work — and it captures the results as it goes: sharpens terminology into CONTEXT.md and records hard-to-reverse decisions as ADRs.
Use it at the start of any non-trivial change. This is the highest-leverage skill in the pack.
Plan
/to-prd
writes: docs/prds/ (or central) + a tracking item
Synthesizes the conversation into a Product Requirements Doc — problem, user stories, modules, testing plan — and opens the tracking epic/issue. No interview; it uses what you've already discussed.
Use it right after grilling to lock the plan down.
/to-issues
Breaks the plan into tracer-bullet vertical slices — thin end-to-end issues, each demoable on its own. For multi-repo work it creates one issue per repo per slice, linked to the parent.
Use it to turn a PRD into grabbable work.
/triage
Moves incoming issues through a state machine — needs-triage, needs-info, ready-for-agent, ready-for-human, wontfix — reproducing bugs and writing durable agent briefs.
Use it to keep the backlog groomed and hand well-specified work to AFK agents.
Build & test
/tdd
Red-green-refactor, one behaviour at a time. Writes a failing unit/integration test, makes it pass, refactors — tests verify behaviour through public interfaces, so they survive refactors.
Use it while implementing a slice.
/functional-test
API tests, Playwright E2E, cross-repo contract tests
The acceptance layer above /tdd: verifies a feature against its acceptance criteria through its real external interface — HTTP APIs, browser flows, and multi-service contract tests. Language-agnostic: it detects the repo's stack.
Use it to prove a feature works end-to-end and to lock cross-service seams.
Review
/review
reads: PRD / acceptance criteria, CONTEXT.md, docs/adr/
Reviews a diff, PR, or branch against the acceptance criteria, the glossary, and the ADRs — flagging correctness, missing tests, architectural smells, naming, and data guardrails by severity (blocker / should-fix / nit) — then posts the review to the configured tracker.
Use it on every change before merge.
/security-review
A focused security pass: secrets, authn/authz (incl. IDOR), input validation & injection, sensitive-data handling, crypto, and dependency risk — findings ranked critical to low, plus an explicit list of what was checked and found safe.
Use it when a change touches auth, external input, data access, or dependencies.
Debug & understand
/diagnose
A discipline for hard bugs: build a fast deterministic feedback loop, reproduce, generate ranked falsifiable hypotheses, instrument one variable at a time, fix with a regression test, then clean up and note what would have prevented it.
Use it for stubborn bugs and performance regressions.
/zoom-out
Asks the agent to go up a layer and map the relevant modules and callers using the project's glossary.
Use it when you're dropped into unfamiliar code.
Improve & hand off
/improve-codebase-architecture
Finds deepening opportunities — shallow modules to consolidate into deep, testable ones — informed by your glossary and ADRs. Presents candidates, then designs interfaces with you.
Use it every few days to fight architectural entropy.
/handoff
Compacts the current conversation into a handoff document (saved to a temp dir, secrets redacted) so a fresh agent can continue.
Use it when context is running low.

Plugins & marketplaces

Like an app store: the marketplace is the store, a plugin is an app, and a skill is a feature inside the app.

Skill
One capability — a folder with a SKILL.md the agent reads when a task matches (e.g. functional-test). On its own, just a file.
Plugin
A bundle you install as a unit — skills plus optional slash commands, subagents, hooks, and MCP servers. Defined by plugin.json. Ours bundles all 14 skills.
Marketplace
A catalog listing one or more plugins and where to find them — how they're discovered, installed, versioned, updated. Defined by marketplace.json. Ours is telus-tools.

How the files fit together

Both definition files live in a .claude-plugin/ folder at the repo root:

repo layout
AI-Claude-engineering-skills/     # the git repo you host
    .claude-plugin/
        plugin.json         # what is this plugin + its skills
        marketplace.json    # what plugins this catalog offers + where
    skills/                 # the actual skills the plugin ships
    templates/  docs/  README.md

plugin.json → { "name": "engineering-skills", "version": "0.1.0", "skills": [ ... ] }. marketplace.json → { "name": "telus-tools", "plugins": [ { "name": "engineering-skills", "source": "./" } ] }. The "source": "./" means the plugin sits at the root of this same repo; a marketplace can also point at other folders or other repos, so one catalog can offer many plugins.

How to create one

(1) Build the plugin — a skills/ folder plus plugin.json. (2) Write marketplace.json listing it. (3) Host it — push the repo to GitHub (public, or private for a team). (4) Version it — bump version in plugin.json when skills change. This repo already has all four.

Why bother vs. copying skill folders

One-command install of the whole bundle, versioned updates (push once, everyone runs update), and central discovery — a private repo becomes your internal "app store".

Install

Option A — as a plugin recommended

The repo already contains .claude-plugin/plugin.json and marketplace.json. Each teammate runs, in Claude Code:

Claude Code
/plugin marketplace add ibrahim-hayeck/AI-Claude-engineering-skills
/plugin install engineering-skills@telus-tools

The marketplace add line uses the owner/repo; the install line uses plugin@marketplace (the name fields inside the two JSON files). One install brings all fourteen skills. Everyone then runs /setup-skills once per repo.

Keeping it updated

After you push changes to the repo, teammates pull the new version:

Claude Code
/plugin marketplace update telus-tools
/plugin                                  # open the plugin manager UI
/plugin uninstall engineering-skills@telus-tools

Option B — drop-in skills (no marketplace)

Copy the skill folders into your skills directory — globally or per-project:

bash
# global (all projects on this machine)
cp -r skills/* ~/.claude/skills/

# or per-project
cp -r skills/* <repo>/.claude/skills/

Keep CONVENTIONS.md and templates/ somewhere the team can find them.

Option C — skills.sh installer (like the original)

If the repo is public, the community installer lets people pick which skills + which agents (Claude Code, Codex, ...):

bash
npx skills@latest add ibrahim-hayeck/AI-Claude-engineering-skills
