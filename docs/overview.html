Engineering Skills

A comprehensive-but-simple set of agent skills that make a coding agent align before it builds, document decisions as it goes, test what it ships, and keep the codebase healthy — the same way across every repo.

What this is

Most AI-coding failures aren’t about typing speed — they’re misalignment (the agent built the wrong thing), verbosity (no shared language), broken code (no feedback loop), and architectural entropy (a growing ball of mud). This pack is a small, composable set of skills that each target one of those failure modes, wrapped in decades-old engineering fundamentals rather than a heavyweight framework.

Skills are invoked as slash commands (e.g. /grill-with-docs). They read a tiny per-repo config that /setup-skills writes once, so the whole team works the same way regardless of language or issue tracker.

The core loop

align→document→plan→build→test→debug→improve

Run /setup-skills once per repo. Then, per change: grill to align, capture a PRD, split into issues, build with tests, debug when stuck, and periodically improve the architecture.

Skills, one by one

Setup · once per repo
/setup-skills
writes → CLAUDE.md · docs/agents/*.md
Records this repo’s issue tracker (GitHub, Jira, or local markdown), where PRDs/ADRs live (in-repo or a central docs repo), the triage label vocabulary, and the CONTEXT.md layout.
Use it once per repo before anything else — every other skill reads what it writes. Re-run only to switch trackers.
Align
/grill-me
Interviews you relentlessly about a plan or design, one question at a time, resolving every branch of the decision tree.
Use it to stress-test any non-code plan before committing to it.
/grill-with-docs
updates → CONTEXT.md · docs/adr/
The same grilling, for code work — and it captures the results as it goes: sharpens terminology into CONTEXT.md and records hard-to-reverse decisions as ADRs.
Use it at the start of any non-trivial change. This is the highest-leverage skill in the pack.
Plan
/to-prd
writes → docs/prds/ (or central) + a tracking item
Synthesizes the conversation into a Product Requirements Doc — problem, user stories, modules, testing plan — and opens the tracking epic/issue. No interview; it uses what you’ve already discussed.
Use it right after grilling to lock the plan down.
/to-issues
Breaks the plan into tracer-bullet vertical slices — thin end-to-end issues, each demoable on its own. For multi-repo work it creates one issue per repo per slice, linked to the parent.
Use it to turn a PRD into grabbable work.
/triage
Moves incoming issues through a state machine — needs-triage → needs-info → ready-for-agent / ready-for-human / wontfix — reproducing bugs and writing durable agent briefs.
Use it to keep the backlog groomed and hand well-specified work to AFK agents.
Build & test
/tdd
Red-green-refactor, one behaviour at a time. Writes a failing unit/integration test, makes it pass, refactors — tests verify behaviour through public interfaces, so they survive refactors.
Use it while implementing a slice.
/functional-test
API tests · Playwright E2E · cross-repo contract tests
The acceptance layer above /tdd: verifies a feature against its acceptance criteria through its real external interface — HTTP APIs, browser flows, and multi-service contract tests. Language-agnostic: it detects the repo’s stack.
Use it to prove a feature works end-to-end and to lock cross-service seams.
Debug & understand
/diagnose
A discipline for hard bugs: build a fast deterministic feedback loop, reproduce, generate ranked falsifiable hypotheses, instrument one variable at a time, fix with a regression test, then clean up and note what would have prevented it.
Use it for stubborn bugs and performance regressions.
/zoom-out
Asks the agent to go up a layer and map the relevant modules and callers using the project’s glossary.
Use it when you’re dropped into unfamiliar code.
Improve & hand off
/improve-codebase-architecture
Finds deepening opportunities — shallow modules to consolidate into deep, testable ones — informed by your glossary and ADRs. Presents candidates, then designs interfaces with you.
Use it every few days to fight architectural entropy.
/handoff
Compacts the current conversation into a handoff document (saved to a temp dir, secrets redacted) so a fresh agent can continue.
Use it when context is running low.

Install

Option A — as a plugin (recommended for a team)

Push this engineering-skills/ folder to an internal git repo (it already contains .claude-plugin/plugin.json and marketplace.json). Then each teammate runs, in Claude Code:

/plugin marketplace add <your-org>/engineering-skills
/plugin install engineering-skills@telus-tools

One install brings all twelve skills. Everyone then runs /setup-skills once per repo.

Option B — drop-in skills (no marketplace)

Copy the skill folders into your skills directory — globally or per-project:

# global (all projects on this machine)
cp -r engineering-skills/skills/* ~/.claude/skills/

# or per-project
cp -r engineering-skills/skills/* <repo>/.claude/skills/

Keep CONVENTIONS.md and templates/ somewhere the team can find them.

Option C — skills.sh installer (like the original)

If the repo is public, the community installer works too and lets people pick which skills + which agents (Claude Code, Codex, …):

npx skills@latest add <your-org>/engineering-skills

Pick your issue tracker at setup
