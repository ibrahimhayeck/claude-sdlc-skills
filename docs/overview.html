4Your first change, start to finish

A real example. The italic line is what you type; below it is what happens.

You type“Help me add an endpoint that returns a customer's orders.”
/grill-with-docs fires. It asks the sharp questions — “Only active orders, or archived too? Paginated? What does an empty result return?” — and records agreed terms in CONTEXT.md. Then: “Clear enough — want me to build it with /tdd?”
You type“Yes, build it.”
/tdd writes one failing test (customer sees their orders newest-first), makes it pass, then the next — one behaviour at a time. When green: “It's a customer-facing endpoint — prove it end-to-end with /functional-test?”
You type“Yes.”
/functional-test writes an API test that hits the real endpoint: an authenticated request returns that customer's active orders, newest-first, and a request with no token is rejected. Green. “Want me to /review before merge?”
You type“Review it.”
/review checks the diff against what you agreed, then asks the safety gate: “This endpoint reads member data — I'll run /security-review before merge.”
You type“Go ahead.”
/security-review finds the real problem: the endpoint never checks that the caller owns the customer — any signed-in user could read anyone's orders (an IDOR). It recommends an ownership check and a failing test for the unauthorized case. You fix it with /tdd; green → merge.

Five plain sentences, and the agent walked the whole loop — proving the feature end-to-end and catching a data-exposure bug before it shipped. You never looked up a command.

6Good to know

Do I have to run setup or create docs?

No. Nothing is required. CONTEXT.md and ADRs are created only if they'd help, and never for trivial changes.

What if I don't know which command to use?

Don't — just describe the task. If nothing fires, type the closest phrase (e.g. "review this") and it will.

Touching auth or data?

/review asks a safety-gate question before merge and runs /security-review when the answer is yes. In data-sensitive work (e.g. anything with PHI/PII) that pass is expected, not optional.

Will it change my code without asking?

It proposes and offers next steps; you approve each hop. The chain guides, it doesn't run away.

Can I make a skill fit how my team works?

Yes — each skill is a single SKILL.md in skills/<name>/. Open it, edit the wording, save. To remove one, delete its folder and its line in plugin.json.

All skills all 13 · one plugin

The complete toolkit in the single engineering-skills plugin, grouped by phase — planning, build & test, review, and architecture, plus the everyday skills above. They all install together; you only invoke what a task needs.

Setup · once per repo
/setup-skills
writes: docs/agents/config.md + a CLAUDE.md pointer
Records this repo's issue tracker (GitHub, Jira, or local markdown), where PRDs/ADRs live, the domain-doc layout, and the test frameworks — all in one config file.
Use it once per repo before anything else — every other skill reads what it writes. Re-run only to switch trackers.
Align
/grill-with-docs
updates: CONTEXT.md, docs/adr/
The same grilling, for code work — and it captures the results as it goes: sharpens terminology into CONTEXT.md and records hard-to-reverse decisions as ADRs.
Use it at the start of any non-trivial change. This is the highest-leverage skill in the pack.
Plan
/to-prd
writes: docs/prds/ (or central) + a tracking item
Synthesizes the conversation into a Product Requirements Doc — problem, user stories, modules, testing plan — and opens the tracking epic/issue. No interview; it uses what you've already discussed.
Use it right after grilling to lock the plan down.
/to-issues
Breaks the plan into tracer-bullet vertical slices — thin end-to-end issues, each demoable on its own. For multi-repo work it creates one issue per repo per slice, linked to the parent.
Use it to turn a PRD into grabbable work.
Build & test
/tdd
Red-green-refactor, one behaviour at a time. Writes a failing unit/integration test, makes it pass, refactors — tests verify behaviour through public interfaces, so they survive refactors.
Use it while implementing a slice.
/functional-test
API tests, Playwright E2E, cross-repo contract tests
The acceptance layer above /tdd: verifies a feature against its acceptance criteria through its real external interface — HTTP APIs, browser flows, and multi-service contract tests. Language-agnostic: it detects the repo's stack.
Use it to prove a feature works end-to-end and to lock cross-service seams.
Review
/review
reads: PRD / acceptance criteria, CONTEXT.md, docs/adr/
Reviews a diff, PR, or branch against the acceptance criteria, the glossary, and the ADRs — flagging correctness, missing tests, architectural smells, naming, and data guardrails by severity (blocker / should-fix / nit); presented in chat, with an option to post to the PR.
Use it on every change before merge.
/security-review
A focused security pass structured on the OWASP Top 10 with compliance hooks (PHI/PII) — access control, crypto, injection, auth, vulnerable deps, logging, SSRF — ranked critical to low, plus what was checked and found safe.
Use it when a change touches auth, external input, data access, or dependencies.
Verify & land
/verify
Evidence-before-claims gate — won't say "done / passing / fixed" without running the check this turn and reading the output. Stops premature success claims.
Use it before any commit, PR, or completion claim.
/finish
Lands a reviewed change: final green verification of the whole branch, confirm the acceptance criteria, then PR / merge / keep / discard, and clean up.
Use it to close out a change cleanly.
Debug & understand
/diagnose
A discipline for hard bugs: build a fast deterministic feedback loop, reproduce, generate ranked falsifiable hypotheses, instrument one variable at a time, fix with a regression test, then clean up and note what would have prevented it.
Use it for stubborn bugs and performance regressions.
/zoom-out
Asks the agent to go up a layer and map the relevant modules and callers using the project's glossary.
Use it when you're dropped into unfamiliar code.
Improve
/improve-codebase-architecture
Finds deepening opportunities — shallow modules to consolidate into deep, testable ones — informed by your glossary and ADRs. Presents candidates, then designs interfaces with you.
Use it on demand when the codebase feels tangled or hard to test.

Who picks the framework — the flexible part

Default is auto-detect — the skills read your manifest and existing tests, and never add a dependency without asking. To pin a choice, set it in docs/agents/config.md (Testing section). Resolution order:

repo config.md▶ org testing-defaults.md▶ auto-detect

So one repo can lock its tools, a set of repos can share defaults via the central docs repo, or you leave everything on auto. No code changes — just edit the config.

Stage by stage

1 · Setup — config.md

/setup-skills writes the pinned frameworks + run commands (or leaves them auto).
Once per repo — everything downstream honours it.

2 · Plan — PRD tests

/grill-with-docs + /to-prd record which behaviours get unit vs functional tests, and the contract seams.
Deliberate coverage tied to acceptance criteria.

3 · Unit — /tdd

Red-green-refactor, one behaviour at a time, through public interfaces.
Fast feedback while building; survives refactors.

4 · End-to-end — /functional-test

API tests, Playwright browser flows, and cross-repo contract tests through the real interface.
Proves the feature works; locks cross-service seams.

5 · Review — /review

Checks acceptance criteria have tests, then asks the auth/data safety gate → /security-review.
No untested behaviour or unreviewed data change reaches merge.

6 · Execute — CI

Runs with the runner from config.md. Units gate every change; E2E gates merges to main; contract tests run on both services.
Regressions caught before deploy.

↺ Debug — /diagnose

Builds a failing test to reproduce a bug, fixes it, leaves a regression test.
The bug can't return; hands back to /review.

All tests + contract files are committed to the repo.
The next iteration/agent runs the same suite; the contract stays the source of truth.

What each file is — and why it stays in the repo

Every file here is committed on purpose. Persisting it is what lets the next iteration — and the next agent — pick up without re-deriving what was already figured out.

CLAUDE.md

The agent's entry point. Its ## Agent skills block says how this repo works and points at docs/agents/.
Why it persists: every future session starts here and inherits the same setup — no re-explaining the repo to each new agent or teammate.

CONTEXT.md

The domain glossary — what each term means, in this repo's language.
Why it persists: keeps humans and agents naming things the same way as the code grows, instead of re-deriving meaning every iteration.

docs/agents/

Per-repo settings the skills read, all in one config.md: issue tracker, PRD/ADR location, domain layout, and test frameworks (pin unit/API/E2E, or leave auto-detect).
Why it persists: written once by /setup-skills; every skill and every future run behaves consistently with zero reconfiguration.

docs/adr/

Short records of hard-to-reverse decisions and the reasoning behind them.
Why it persists: stops a later iteration from re-litigating or accidentally "fixing" a deliberate choice — the why outlives the people who made it.

PRD & .scratch issues

What we're building and the vertical slices to build it — the plan and its acceptance criteria.
Why it persists: the durable record of intent; future work links back to the epic, and reviews and tests are measured against it.

tests & contracts

Executable proof the behaviour works, plus the cross-service contract both sides test against.
Why it persists: guards every future change against regressions, and the contract stays the shared source of truth as services evolve independently.
Two swaps to know: pick GitHub/Jira in /setup-skills and every .scratch/ORD-42/* file becomes a remote GitHub issue / Jira story instead (one per repo, linked to the epic) — nothing else moves. Choose in-repo docs and the PRD/cross-repo ADR move into each repo's docs/ rather than the hub (best only for single-repo work). temp = the /handoff doc goes to your OS temp dir and debug logs are deleted at the end — deliberately not persisted.

Reference — plugins & marketplaces

Secondary: how the packaging works and what's in the repo. Skip unless you're curious or maintaining the plugin.

Like an app store: the marketplace is the store, a plugin is an app, and a skill is a feature inside the app.

Skill
One capability — a folder with a SKILL.md the agent reads when a task matches (e.g. functional-test). On its own, just a file.
Plugin
A bundle you install as a unit — skills plus optional slash commands, subagents, hooks, and MCP servers. Defined by plugin.json. Ours bundles all 13 skills.
Marketplace
A catalog listing one or more plugins and where to find them — how they're discovered, installed, versioned, updated. Defined by marketplace.json. Ours is engineering-tools.

How the files fit together

Both definition files live in a .claude-plugin/ folder at the repo root:

repo layout
claude-sdlc-skills/     # the git repo you host
    .claude-plugin/
        plugin.json         # what is this plugin + its skills
        marketplace.json    # what plugins this catalog offers + where
    skills/                 # the actual skills the plugin ships
    templates/  docs/  README.md

plugin.json → { "name": "engineering-skills", "version": "0.1.0", "skills": [ ... ] }. marketplace.json → { "name": "engineering-tools", "plugins": [ { "name": "engineering-skills", "source": "./" } ] }. The "source": "./" means the plugin sits at the root of this same repo; a marketplace can also point at other folders or other repos, so one catalog can offer many plugins.

How to create one

(1) Build the plugin — a skills/ folder plus plugin.json. (2) Write marketplace.json listing it. (3) Host it — push the repo to GitHub (public, or private for a team). (4) Version it — bump version in plugin.json when skills change. This repo already has all four.

Why bother vs. copying skill folders

One-command install of the whole bundle, versioned updates (push once, everyone runs update), and central discovery — a private repo becomes your internal "app store".
