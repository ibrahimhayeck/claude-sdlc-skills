# Review

Review a change before it merges. No setup required.

## 1. Get the change
In order of preference: a PR reference passed as an argument → fetch it; the current branch
vs its base (`git diff <base>...HEAD`); or staged/unstaged changes (`git diff`). If there's
an issue/PRD linked, read it and review against its acceptance criteria; if not, review
against what the change clearly intends to do.

## 2. Build just enough context
Skim the surrounding code so you judge whether the change fits how things work. If the repo
has a `CONTEXT.md`, use its vocabulary and flag naming that drifts from it. If it has
`docs/adr/`, don't "correct" a decision that an ADR deliberately made — call it out instead.

## 3. Review against this checklist
Group findings by severity — **blocker / should-fix / nit** — each with what, where
(describe by symbol/behaviour, not brittle line numbers), and a concrete fix.

- **Correctness** — does it do what it intends? Edge cases, error paths, null/empty, concurrency.
- **Tests** — is the new behaviour covered, and are the tests behavioural (through the public
  interface) rather than coupled to implementation?
- **Clarity & naming** — names reveal intent; matches the glossary if there is one.
- **Architecture** — duplication, coupling, or a shallow module introduced; flag it, don't demand a rewrite.
- **Safety** — no secrets or personal/production data in code, logs, tests, or fixtures; for
  changes touching auth, external input, or dependencies, note that a deeper security pass is warranted.
- **Scope** — does only what's intended; call out unrelated drive-by edits.

## 4. Deliver
Print the review in chat, structured as below. If the change is a GitHub PR and `gh` is
available, offer to post it as a PR comment (prefixed `> *Generated by AI during review.*`).

```markdown
## Review summary
Verdict: approve / approve-with-nits / changes-requested — and why, in one paragraph.

## Blockers
- ...
## Should-fix
- ...
## Nits
- ...
```

Be specific and kind. If there are no blockers, say so plainly; note what's genuinely good
without padding.

## Next
Route from the findings, and offer the next step:
- a blocker with a clear fix → `/tdd` to fix it test-first;
- a mysterious or intermittent bug → `/diagnose`;
- no blockers → it's good to merge.

After a fix lands, offer to re-run `/review` on the updated change so nothing regressed.
