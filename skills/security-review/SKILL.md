# Security Review

A security-focused pass over a change. Narrower and deeper than `/review` — assume an
attacker and look for what could go wrong. Grounded in the per-repo config (`/setup-skills`
if missing) and any compliance constraints noted in the PRD/ADRs.

Any comment posted to the tracker **must** start with:

```
> *This was generated by AI during security review.*
```

## 1. Gather the change
Get the diff (PR/branch/staged, same as `/review`). Note what the change touches:
authentication, authorization, user input, data access, file/network I/O, crypto,
serialization, or dependencies — those are where the risk concentrates.

## 2. Threat-model the change
For the touched surface, ask: what's the untrusted input, what's the trust boundary it
crosses, and what's the worst thing an attacker could do with it?

## 3. Review against this checklist
Group findings by severity (**critical / high / medium / low**), each with the concrete risk
and a fix.

- **Secrets** — no hardcoded keys, tokens, passwords, connection strings; nothing sensitive
  in logs, error messages, traces, or fixtures. Secrets come from env/secret store.
- **AuthN / AuthZ** — every new endpoint/action checks identity *and* permission; no missing
  ownership checks (IDOR), no trusting client-supplied roles/IDs.
- **Input validation & injection** — untrusted input is validated and parameterized: SQL/NoSQL
  injection, command injection, path traversal, SSRF, XSS, unsafe deserialization, template
  injection. Prefer allow-lists.
- **Sensitive data** — PII/PHI/financial data is minimized, encrypted in transit and at rest
  where required, not logged, and handled per the compliance notes in the PRD/ADRs.
- **Dependencies** — new/updated packages are reputable and pinned; flag known-vulnerable or
  abandoned ones and suggest running the ecosystem audit (`npm audit`, `pip-audit`, etc.).
- **Crypto & tokens** — standard libraries only (no home-rolled crypto), sound randomness,
  sensible token lifetimes and storage.
- **Errors & rate limits** — failures don't leak internals; sensitive actions are rate-limited.

## 4. Deliver
Post one security review to the configured tracker, structured as:

```markdown
> *This was generated by AI during security review.*

## Security review summary
Overall risk: none / low / medium / high, and the headline reason.

## Critical / High
- Risk → impact → fix

## Medium / Low
- ...

## Checked and OK
- Short list of what you verified is safe (so reviewers know it was looked at).
```

If you can't fully assess something (e.g. auth flows spanning services you can't see), say so
and name what a human should verify. Don't imply a change is secure beyond what you reviewed.
