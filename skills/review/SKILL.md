# Review

Review a change before it merges. Grounded in the same per-repo config the other skills use
(run `/setup-skills` if `docs/agents/` is missing): it reads the tracker, the PRD/issue, the
domain glossary, and the ADRs so the review speaks the project's language and doesn't
re-litigate settled decisions.

Any review comment posted to the tracker **must** start with:

```
> *This was generated by AI during review.*
```

## 1. Gather the change
Get the diff to review, in this order of preference:
- a PR/MR reference or issue key passed as an argument → fetch it from the tracker;
- the current branch vs its base (`git diff <base>...HEAD`);
- staged/unstaged changes (`git diff`), if that's what the user means.
Also fetch the linked **PRD / issue** and its **acceptance criteria** — the review is
measured against them, not against your own idea of the feature.

## 2. Build context
- Read `CONTEXT.md` (glossary) so you flag naming that drifts from the ubiquitous language.
- Read ADRs in the touched area. If the change contradicts an ADR, say so and link it;
  don't silently "correct" a deliberate decision.
- Explore the surrounding code enough to judge whether the change fits how things work.

## 3. Review against this checklist
Report findings grouped by severity (**blocker / should-fix / nit**). For each: what, where
(describe the location by symbol/behaviour, not brittle line numbers), and a concrete suggestion.

- **Correctness** — does it actually meet every acceptance criterion? Edge cases, error
  paths, off-by-one, null/empty, concurrency.
- **Tests** — is each acceptance criterion covered? Are the tests behavioural (through the
  public interface) rather than coupled to implementation? Hand deeper testing gaps to
  `/tdd` or `/functional-test`.
- **Architecture** — shallow modules, leaky seams, duplication, or coupling introduced. Use
  the vocabulary from `improve-codebase-architecture` (module, interface, seam, depth). Flag
  candidates for `/improve-codebase-architecture` rather than demanding a rewrite in review.
- **Naming & clarity** — matches `CONTEXT.md`; names reveal intent.
- **Data & safety guardrails** — no real/production data in tests or fixtures, no secrets or
  PII in code/logs. For anything touching auth, external input, or dependencies, recommend a
  `/security-review`.
- **Scope** — the change does what the issue says and no more; unrelated drive-by edits are
  called out.

## 4. Deliver
Post a single review to the tracker (PR/MR review comment, Jira comment, or a note in the
local `.scratch/` issue — per `docs/agents/issue-tracker.md`), structured as:

```markdown
> *This was generated by AI during review.*

## Review summary
One-paragraph verdict: approve / approve-with-nits / changes-requested, and why.

## Blockers
- ...

## Should-fix
- ...

## Nits
- ...

## Acceptance criteria
- [x] Criterion met
- [ ] Criterion NOT met — reason
```

Be specific and kind. Praise what's genuinely good; don't pad. If there are no blockers,
say so plainly.

## Next step (core chain)
**Always ask the safety gate before merge**, even on a clean review:

> "Does this change touch authentication/authorization, personal or production data,
> external input, or third-party dependencies?"

If the answer is yes — or the developer is unsure — run `/security-review`. In regulated or
data-sensitive work (e.g. anything handling PHI/PII), treat that as expected, not optional.

Then route from the findings and offer the next step: a blocker with a clear fix → `/tdd` to
fix it test-first; a mysterious or intermittent bug → `/diagnose`; a deeper structural issue →
`/improve-codebase-architecture`; behaviour not yet proven end-to-end → `/functional-test`.
No blockers and the safety gate is clear → hand to `/verify` (evidence gate) and then
`/finish` to land the change. After any fix lands, offer to re-run `/review`.
